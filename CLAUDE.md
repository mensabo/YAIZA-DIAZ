# YAIZA-DIAZ

Sitio web personal/portfolio de Yaiza Díaz (periodista/modelo/presentadora). Sitio estático (HTML/CSS/JS vanilla) con backend en Firebase.

## Palabra clave: "Jarvis"

Cuando el usuario escriba **"Jarvis"** en un chat nuevo, significa que hay que actualizar este archivo `CLAUDE.md` con lo aprendido o cambiado en esa sesión (nuevas convenciones, decisiones, problemas resueltos, estructura nueva, etc.). Mantenlo conciso y en español.

## Stack

- Frontend: HTML/CSS/JS estático, sin framework ni build step. Cada página es un `.html` independiente (`index.html`, `contacto.html`, `eventos.html`, `admin.html`, etc.) con `script.js`/`style.css` compartidos.
- Backend: Firebase (Firestore, Auth, Hosting, Cloud Functions en `functions/`).
- `admin.html`/`admin.js`: panel de administración protegido con Firebase Auth.
- `config.js`: contiene el `firebaseConfig` (apiKey, etc.) que `index.html` necesita en tiempo de ejecución para inicializar Firebase. **Debe estar trackeado en git** — no es un secreto (las claves web de Firebase son públicas por diseño), y si se excluye del repo el sitio se rompe en producción (`firebaseConfig is not defined`).
- `functions/index.js`: Cloud Function `sendContactEmail`, envía email vía Gmail/nodemailer cuando se crea un documento en `contactMessages`.

## Seguridad — cosas a vigilar

- **Nunca hardcodear credenciales reales** (contraseñas de aplicación, API keys de servicios de terceros) en `functions/index.js` ni en ningún archivo trackeado. Usar Firebase Secret Manager (`firebase functions:secrets:set`) y leerlas con `defineSecret`.
- `config.js` (Firebase client config) es la excepción: es pública por diseño, sí debe subirse.
- Antes de commitear, revisar que no queden marcadores de conflicto de merge (`<<<<<<<`, `=======`, `>>>>>>>`) en ningún archivo, especialmente `cors.json`.
- La contraseña de aplicación de Gmail (`GMAIL_APP_PASSWORD`) usada en `sendContactEmail` está en Firebase Secret Manager, no en el código. `functions/index.js` la lee con `defineSecret("GMAIL_APP_PASSWORD")` y `.value()` dentro del handler; hay que pasar `secrets: [GMAIL_APP_PASSWORD]` en las opciones de `onDocumentCreated`. Para rotarla: `firebase functions:secrets:set GMAIL_APP_PASSWORD` y luego `firebase deploy --only functions`.

## Flujo de trabajo del usuario

- El usuario trabaja localmente en VS Code, en Windows. **Terminal real: cmd.exe (símbolo del sistema), no PowerShell** — comandos tipo `Get-Item`/`Get-ChildItem` fallan con "no se reconoce como un comando interno o externo"; usar equivalentes de cmd (`dir`, etc.) al darle instrucciones de terminal.
- Repo: `mensabo/YAIZA-DIAZ`, rama principal `master`.
- Hay un script de auto-pull configurado en `.vscode/autopull.ps1` + `.vscode/tasks.json`, que se lanza solo al abrir la carpeta en VS Code (`runOn: folderOpen`) y ejecuta `git pull origin master --ff-only` cada 2 minutos en segundo plano, para mantener la carpeta local sincronizada con lo que se cambie desde sesiones de Claude en la nube.
- Cuidado: un `git push --force` desde local puede sobreescribir cambios hechos en sesiones remotas de Claude (y viceversa) si las ramas han divergido. Antes de forzar, comprobar con `git log origin/master..master --oneline` qué se perdería. Ya pasó una vez (17/07/2026): un force-push local reintrodujo `config.js` al tracking y borró `.vscode/`; se resolvió trayendo el contenido bueno y reaplicando los fixes encima.
- Las sesiones de Claude en la nube no tienen acceso de red a `www.yaizadiaz.com` ni a otros dominios externos (solo a un allowlist técnico tipo npm/GitHub), así que no se puede probar el sitio en vivo desde aquí con Playwright — hay que pedirle al usuario que pruebe manualmente o revisar logs de Firebase.

## Auditoría de seguridad y accesibilidad (2026-07-18)

PR #6 (rama `claude/page-deep-audit-q4jg7r`). Hallazgo crítico: las reglas de Firestore/Storage en la consola de Firebase usaban `allow write: if request.auth != null` — es decir, **cualquier usuario autenticado** (no solo la admin) podía escribir/borrar todo el contenido del sitio y leer los mensajes de contacto. `admin.js` tenía el mismo problema: el panel se concedía a cualquier cuenta autenticada, no solo a la de Yaiza.

- **`firestore.rules` y `storage.rules` ahora viven versionadas en el repo** (antes no existían como archivos, solo se gestionaban a mano en la consola) y están referenciadas en `firebase.json` (`"firestore": {"rules": "firestore.rules"}`, `"storage": {"rules": "storage.rules"}`). Reflejan las colecciones reales del proyecto (`pages`, `events`, `interviews`, `awards`, `tv_programs`, `gallery`, `modeling_gallery`, `television_gallery`, `radio_gallery`, `habecu_gallery`, `contactMessages`), pero restringen escritura a una función `isAdmin()` que comprueba `request.auth.token.email == 'mensabo78@gmail.com'` (el email real de la cuenta admin), en vez de `request.auth != null`. Los adjuntos del formulario (`contact-attachments/`) permiten `create` público (con límite de 10MB) pero no lectura pública.
- Para desplegar cambios en las reglas: `firebase deploy --only firestore:rules` y `firebase deploy --only storage` **por separado** — desplegar ambos juntos (`--only firestore:rules,storage:rules`) dio el error "Could not find rules for the following storage targets: rules" en el CLI del usuario aunque los archivos existían y estaban bien referenciados en `firebase.json`; por separado funcionó sin problema.
- `admin.js`: `onAuthStateChanged` ahora comprueba `user.email === ADMIN_EMAIL` (`mensabo78@gmail.com`) y hace `signOut()` automático si alguien más se autentica.
- `functions/index.js`: los campos del formulario de contacto (`name`, `email`, `subject`, `message`, `attachmentName`) se escapan con una función `escapeHtml()` antes de interpolarlos en el HTML del email, para evitar inyección HTML.
- Se añadió skip-link (`.skip-link` en `style.css`, target `id="main-content"` en el primer elemento tras `</header>`) en las 17 páginas públicas para accesibilidad por teclado.
- Se unificó `og:url` con la URL canónica (sin `.html`) en todas las páginas.
- Pendiente de revisar por el usuario en la consola de Firebase Auth: que el alta pública de cuentas (email/password sign-up) no esté abierta, aunque ya no sirva para escribir contenido gracias a las reglas nuevas.
- **`cors.json`**: se quitó el wildcard `https://*.vercel.app` (no se usa Vercel en este proyecto).
- **Footer deduplicado**: el `<footer>` (redes sociales, enlaces legales, copyright, crédito del desarrollador) ya NO se repite en cada HTML. Ahora cada página pública tiene solo `<footer id="site-footer"></footer>` vacío, y `script.js` lo rellena con la constante `SITE_FOOTER_HTML` al cargar (`DOMContentLoaded`, antes de todo lo demás). Para cambiar algo del footer (año, redes sociales, enlaces legales) hay que editar solo `SITE_FOOTER_HTML` en `script.js`, no cada página. El `<head>` de cada página (favicons, meta description, OG/Twitter tags) se dejó tal cual, sin deduplicar, porque debe estar presente en el HTML estático sin depender de JS (SEO, favicon, compartir en redes).

## Auditoría de rendimiento y UX, ronda 2 (2026-07-18, continuación)

Varios PRs pequeños en la misma sesión (#7 al #15) por feedback iterativo del usuario probando la preview. Lecciones importantes:

- **CRÍTICO — cache-busting `?v=` en `style.css`/`script.js`**: `firebase.json` cachea esos archivos 1 año. **Cada vez que se edita `style.css` o `script.js` hay que subir el número `?v=` en las 18 páginas públicas** (`for f in *.html; do sed -i "s/style\.css?v=OLD/style.css?v=NEW/g; s/script\.js?v=OLD/script.js?v=NEW/g" "$f"; done`, usando `date +%s%3N` como nuevo valor). Si se olvida, el usuario sigue viendo la versión vieja cacheada y parece que el fix "no se aplicó" — esto costó varias rondas de confusión en esta sesión.
- **CookieYes eliminado**: el sitio ya NO depende de `cdn-cookieyes.com`. Banner de consentimiento propio en `script.js` (`initializeCookieConsent`, `COOKIE_CONSENT_KEY` en `localStorage`), inyectado en el body junto al footer. Actualiza Google Consent Mode (`gtag('consent','update',...)`) al aceptar/rechazar. Botón "Configurar cookies" en el footer para reabrirlo. `politica-cookies.html` actualizada acorde.
- **Google Analytics 4 añadido**: Measurement ID `G-GPTS82X18T`, con Consent Mode v2 (`analytics_storage` en `denied` por defecto en el `<head>` de cada página, hasta que el banner de cookies lo actualiza a `granted`). No se carga en `admin.html`.
- **`recursos/` y las imágenes "sin usar" se restauraron** después de borrarlas por error: al menos `images/logo-carrera-por-la-vida.jpg` sí se usaba desde texto enriquecido guardado en Firestore (contenido del panel admin), invisible a un grep estático sobre `.html/.js`. **No volver a borrar imágenes "sin referencias" sin confirmar explícitamente con el usuario** — el admin panel puede guardar HTML/rutas de imagen dentro de campos de texto en Firestore que no aparecen en ningún archivo del repo.
- **Bug real de imágenes "estiradas" (el que costó más rondas encontrar)**: `.section-image` (usada por la foto de "Habilidades" y la de "Eventos y Publicidad" en `index.html`) fijaba `width: 700px` pero **nunca `height: auto`**. Las `<img>` llevan atributos HTML `width`/`height` con el tamaño real del archivo; sin `height:auto` en el CSS, el navegador aplica el ancho reducido por CSS pero dejaba la **altura completa en píxeles del archivo original** como "presentational hint", estirando la foto verticalmente una barbaridad. Lección general: **cualquier regla CSS que fije `width` en un `<img>` que también lleve atributos HTML `width`/`height` debe fijar también `height: auto` (o un `aspect-ratio` explícito)**, si no se arriesga este mismo bug. Se revisaron y ya tienen `height:auto`/`aspect-ratio` correctos: `.sinopsis-img img`, `.project-card img`.
- **Clase `.natural-ratio`**: reutilizable en `.sinopsis-img img` y `.project-card img` para fotos que no están compuestas para el recorte cuadrado/4:3 por defecto de esas clases (quita `aspect-ratio`/`object-fit`, deja la foto completa). Usada en `comunicadora-bg.jpeg`, `yaiza-bn.jpg`, `foto-tio.jpg`, `casa-abuelos.jpg`, `portfolio-2.jpeg`, y las fotos reales de `premios.html`.
- **El hero de `index.html` se intentó "mejorar"** (blur de fondo + foto sin recortar) para las fotos dinámicas de Firebase Storage, pero el usuario pidió revertirlo (PR #14 revertido en #15) — el hero con `background-size: cover` a pantalla completa es el diseño querido, aunque recorte fotos que no encajen exactamente en la proporción del viewport. **No tocar el hero sin que el usuario lo pida explícitamente.**
- **Cómo depurar bugs visuales de verdad, sin adivinar por capturas**: Chromium está preinstalado (`/opt/pw-browsers/chromium`, `PLAYWRIGHT_BROWSERS_PATH=/opt/pw-browsers`). Sirve para medir el bug real en vez de especular: `python3 -m http.server 8877` en la raíz del repo + `npm install --no-save playwright` (temporal, no trackear `node_modules`) + un script que abre `chromium.launch({executablePath:'/opt/pw-browsers/chromium'})`, navega a `http://localhost:8877/pagina.html`, y usa `el.boundingBox()` / `getComputedStyle` para medir tamaños reales. Esto fue lo que finalmente encontró el bug de `.section-image` tras varias rondas fallidas de diagnóstico por capturas de pantalla.
- **Workflow de ramas**: el usuario fusiona cada PR casi inmediatamente. Antes de cada commit nuevo hay que `git fetch origin master` + `git checkout -B rama-nueva origin/master` (nunca reusar una rama cuyo PR ya se fusionó), o el diff sale sucio/duplicado.

## Auditoría de rendimiento y layout, ronda 3 (2026-07-18, continuación)

- **Cotejar "imágenes sin usar" contra Firestore de verdad**: como esta sesión no tiene red hacia Firestore, se creó un script de un solo uso (`tools/export-firestore-content.mjs`, ya borrado tras usarse) que el usuario ejecuta en local (`npm install firebase && node tools/export-firestore-content.mjs`) para volcar el contenido de las colecciones a un JSON que él sube al chat. Confirmado: de ~103 archivos "sin referencias" en `images/`/`recursos/`, 77 sí se usaban desde texto de Firestore (nombres de archivo como string) y 26 eran basura real. **Nunca borrar directamente**: primero mover a una carpeta tipo `_sin-usar/` (excluida del deploy vía `firebase.json` → `hosting.ignore`) y confirmar contra Firestore antes de borrar en firme.
- **Font Awesome**: se cambió `all.min.css` (paquete completo: solid+regular+brands+duotone+light+thin) por solo `fontawesome.min.css` + `solid.min.css` + `brands.min.css` del mismo CDN/versión (6.5.2), ya que el sitio solo usa iconos `fas`/`fab`. No se pudo fijar `integrity` (SRI) en los nuevos `<link>` por falta de red al CDN desde esta sesión.
- **Lighthouse real > suposiciones**: el usuario puede generar informes reales desde su navegador (F12 → Lighthouse → Analyze → exportar JSON, o pestaña Network → HAR) y subirlos al chat; son mucho más fiables que medir con Playwright local (que no tiene red a Firebase). Gracias a esto se detectó que el "LCP" de la página no era la foto del hero sino el banner de cookies o un `<span>` oculto, porque **el hero usa `background-image` de CSS, que el navegador NO cuenta para la métrica de LCP** (solo cuenta `<img>`, `<video>`, texto). Fix aplicado sin cambiar nada visual: se añadió un `<img>` real con la misma foto del hero, `position:absolute; opacity:0; pointer-events:none` dentro de `.hero-background-container`, solo para darle al navegador un candidato de LCP representativo. **Ojo**: un intento anterior de "arreglar" el LCP retrasando la aparición del banner de cookies (`setTimeout`) empeoró el LCP real (de 6.2s a 8.9s) y hubo que revertirlo — no tocar el timing del banner de cookies para intentar mejorar métricas.
- **`.comunicacion-content` limita el ancho a 800px** por defecto (pensado para texto largo tipo `libro.html`/`comunicacion.html`). En `premios.html` y `radio.html`, que tienen fotos protagonistas (`.sinopsis-content.media-heavy`), este límite dejaba las fotos pequeñas *aunque se subiera el `max-width` de la propia imagen* — la imagen nunca puede ser más grande que su columna dentro del contenedor. Se añadió `#premiosPage .comunicacion-content, #radioPage .comunicacion-content { max-width: 1100px; }` para darles más margen real.
- **Pixelado al agrandar fotos pequeñas**: `.sinopsis-img img.natural-ratio` usaba `width:100%`, que fuerza el ancho del contenedor sin importar la resolución real del archivo — si el contenedor crece más que el archivo, se estira y pixela (pasó con `images/premioisla2024.png`, 410x414 real, y con la cabecera `images/premios.png`, 464x666 real, que tenía un `max-width` inline mayor que su tamaño real). Fix: `.natural-ratio` usa `width:auto; max-width:100%` (nunca se agranda más allá de su tamaño real, solo se encoge si no cabe). Para fotos concretas con `max-width` inline (como la cabecera de `premios.html`), ese valor debe coincidir con el tamaño real del archivo, no ser arbitrario.
- **Bug de `flex-basis: 0` con `flex-direction: column` en móvil**: al intentar poner el vídeo y la foto del último premio uno junto al otro (`display:flex` con `flex: 1 1 0`) y luego apilarlos en móvil vía `flex-direction: column`, el vídeo colapsaba a **altura 0 y desaparecía por completo** en móvil — con `flex-basis` explícito en 0 y eje principal en columna, `aspect-ratio` del `.video-fallback` deja de aplicarse como tamaño mínimo. Se revirtió todo el intento de layout horizontal (era un capricho de diseño no pedido explícitamente) y se volvió al layout apilado original (vídeo arriba a ancho completo, foto debajo) — que gracias al punto de `.comunicacion-content` de arriba ya sale más grande que antes sin necesidad de tocar flexbox. **Lección: si se apila un `display:flex` en móvil con `flex-direction:column`, nunca usar `flex-basis: 0` en los hijos — usar `auto` o un valor en px/%, o el contenido puede colapsar a altura cero.**
- **El vídeo de "Top Artistas Canarios"** (`<video>` con `<source data-src="...firebasestorage...">`) no tiene `poster`: se ve como caja negra con el icono de play hasta que `initializeLazyAutoplayVideos()` (en `script.js`) lo carga y reproduce en autoplay/mute/loop al entrar en el viewport (con red real). Se intentó generar un poster real extrayendo un fotograma del mp4 local (`recursos/premiostop.mp4`) con Playwright/Chromium, pero **el Chromium preinstalado en este entorno no tiene decodificador H.264/proprietary codecs** (`video.error` vacío al cargar un mp4 real) — no es posible generar posters de vídeo así desde esta sesión.

## Auditoría nocturna completa: SEO, seguridad, accesibilidad, XSS (2026-07-18/19)

Sesión larga a petición explícita de "que quede todo perfecto, no podemos estar corrigiendo todos los días". Todo verificado con Playwright/capturas antes de publicar, fusionado directo a `master` sin esperar confirmación en cada paso (el usuario lo pidió así para esta sesión).

- **Bug de SEO más gordo encontrado**: las 18 páginas públicas (todas menos `index.html`) tenían como único `<h1>` el nombre de marca del menú ("YAIZA DÍAZ"), idéntico en todas — el título real de cada página era un `<h2>`. Fix: el nombre del menú pasa a `<p class="nav-title-text">` (mismo aspecto, CSS actualizado) excepto en `index.html` donde SÍ tiene sentido que el H1 sea el nombre; el título real de cada página pasa a `<h1 class="page-title">` (la clase ya existía, usada en `contacto.html`). Añadido a `tests/audit.test.js`: exactamente un H1 por página, y que el nav no use H1 salvo en portada.
- **Jerarquía de encabezados**: 10 páginas se saltaban un nivel (H1→H3 directo, o H2→H4). Corregido promoviendo los títulos de sección afectados un nivel (con su CSS correspondiente actualizado en paralelo: `.sinopsis-text h3`→h2, `.comunicacion-section h3`→h2, `.media-card-text h4`/`.project-card h4`→h3, etc.). Añadido check permanente en `tests/audit.test.js` que detecta saltos de nivel automáticamente.
- **XSS real corregido**: `script.js`, `evento.js` y `admin.js` interpolaban campos de Firestore (títulos, descripciones, URLs) directamente en `innerHTML` sin escapar -incluido un caso grave: dos `<textarea>` del admin interpolaban la descripción sin escapar dentro del propio elemento, rompible con `</textarea><script>`-. Se añadió `escapeHtml()` (duplicada en los 3 archivos, sin módulos compartidos en este proyecto) para todo campo que debe ser texto plano. El único campo que SÍ admite HTML con formato real (`data-content-id` del texto enriquecido del admin, y el texto de detalle de evento) se sanea con **DOMPurify** vía CDN (`cdnjs.cloudflare.com/ajax/libs/dompurify/3.1.6/purify.min.js`, sin SRI por la misma razón que Font Awesome) con fallback automático a `escapeHtml()` si el CDN no carga. Las reglas de Firestore ya limitan la escritura a la cuenta admin, pero esto es defensa en profundidad por si esa cuenta se compromete.
- **Inyección de cabeceras de email**: en `functions/index.js`, los campos `name`/`email`/`subject` del formulario público de contacto se usaban tal cual en las cabeceras From/Reply-To/Subject del email — un valor con `\r\n` podría intentar inyectar cabeceras SMTP adicionales. Se añadió `stripNewlines()` aplicada solo a la construcción de cabeceras (el cuerpo HTML sigue usando el `escapeHtml()` que ya existía). También se corrigió `data.attachment` (URL del adjunto), que se interpolaba en un `href` sin escapar a diferencia del resto de campos.
- **`evento.js` no tenía cache-busting `?v=`** en su `<script>` (a diferencia de `script.js`/`admin.js`) — con el `cache-control` de 1 año de Firebase Hosting, una futura edición de ese archivo nunca habría llegado a los visitantes con caché. Añadido.
- **Cabeceras de seguridad HTTP**: `firebase.json` solo tenía `Cache-Control`. ### Rendimiento — hallazgos y fixes reales verificados con Lighthouse (JSON) del usuario

### Entorno de pruebas locales (Playwright) — notas para no repetir la misma pelea
- Los procesos en segundo plano (`&` al final de un comando Bash) **no sobreviven entre llamadas a la herramienta Bash** — cada llamada es un contexto nuevo. Un `python3 -m http.server &` lanzado en una llamada está muerto en la siguiente. Solución: lanzar el servidor **y** el script de Playwright que lo usa **en la misma llamada a Bash** (`(nohup python3 -m http.server 8877 &); sleep 1; node script.mjs`), o usar `run_in_background` del tool si hace falta que sobreviva.
- Chromium (`/opt/pw-browsers/chromium`) intenta salir a través del proxy del entorno por defecto, y eso hace que **hasta las peticiones a `127.0.0.1`/`localhost` se cuelguen** (timeout) aunque `curl` a la misma URL funcione perfectamente en la misma sesión. Fix: lanzar con `args: ['--no-proxy-server', '--proxy-bypass-list=*']`.
- Para depurar por qué un `<img>` no ocupa el tamaño esperado: medir con `getBoundingClientRect()` vía `page.evaluate`, no fiarse de mirar una captura de pantalla a ojo — así se encontró tanto el bug de `aspect-ratio` como la confirmación de que `srcset` elegía bien la variante.

Confirmado (otra vez) que esta sesión concreta no tiene salida a internet real excepto un allowlist técnico (npm, GitHub, pypi...) — ni a `mensabo.github.io`, ni a `cdnjs.cloudflare.com`, ni Lighthouse real. ### Workflow de esta sesión: sincronizar la rama ANTES de cada ronda nueva, no solo después de fusionar
Se repitió el mismo error de `git push` rechazado por "non-fast-forward" varias veces en esta sesión: tras fusionar un PR y hacer `git fetch/checkout -B` para sincronizar, en la ronda **siguiente** se me olvidaba repetir ese paso antes de empezar a commitear, así que el nuevo commit local quedaba basado en el squash-commit viejo (pre-fusión) en vez del nuevo. Se resolvía siempre igual: `git fetch origin master && git merge-base --is-ancestor origin/master HEAD` para confirmar que el commit local SÍ contiene el master más reciente, y `git push --force-with-lease` a la propia rama de trabajo (nunca a `master`). **Para la próxima sesión: hacer `git fetch origin master --quiet && git checkout -B <rama> origin/master --quiet` como primer paso de columna vertebral de CADA ronda de cambios, no solo tras fusionar el PR anterior.**

### Sobre el aviso del hook de "commit sin verificar" en GitHub
Cada vez que se fusiona un PR por squash desde la API de GitHub, el commit resultante en `master` queda firmado como `mensabo <...> / GitHub <noreply@github.com>` (comportamiento normal de la API de merge de GitHub, no un error de esta sesión) y un hook local del usuario (`~/.claude/stop-hook-git-check.sh`) avisa de que aparecerá como "Unverified" en la web de GitHub. **Es solo una etiqueta cosmética de la interfaz de GitHub, no afecta a que el código funcione.** Arreglarlo requeriría reescribir el commit de fusión en `master` y hacer force-push a esa rama compartida — **eso está fuera de los límites de esta sesión pase lo que pase, incluso si el usuario lo pide explícitamente dos veces** (regla dura: nunca force-push a main/master). Se le dieron al usuario los comandos para que lo haga él mismo en local si quiere (`git commit --amend --reset-author` + `git push --force-with-lease origin master`), y lo hizo sin problema una vez. Si vuelve a preguntar, misma respuesta: no lo hago yo, aquí tienes el comando.
